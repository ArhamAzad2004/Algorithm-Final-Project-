💡 Key Concepts

▼
🔑 Rolling Hash — O(1) per window slide

Instead of recomputing the hash of each window from scratch (O(m)), we update it in O(1): remove the contribution of the outgoing character and add the incoming character. This is the core trick that makes Rabin-Karp efficient.

🔑 Spurious Hit — a hash collision

When hash(window) = hash(pattern) but the actual characters are different. This is unavoidable with any finite modulus. We must always verify character-by-character on a hash match. Spurious hits add O(m) verification cost each.

🔑 Why use a large prime q?

Mod a prime distributes hash values more uniformly. A larger prime means fewer collisions. With q=101, collisions happen ~1/101 times on average. With q=1000003, ~1/1000003 times. A power of 2 as modulus would be far worse.

🔑 Expected O(|T| + |P|) time

|T|−|P|+1 hash comparisons at O(1) each = O(|T|). Initial hash computation = O(|P|). Verification only triggers on (rare) hash matches. Expected total = O(|T| + |P|). Worst case O(|T|×|P|) if every window is a spurious hit.

🔑 Rabin-Karp vs KMP vs Naive

Naive: O(|T|×|P|) worst case, no preprocessing. KMP: O(|T|+|P|) guaranteed, O(|P|) preprocessing (failure table). Rabin-Karp: O(|T|+|P|) expected, O(|P|) preprocessing (hash). KMP is deterministic; Rabin-Karp is probabilistic but simpler to implement and extendable to 2D matching.

📖 Lecture Notes — Rabin-Karp String Matching

▼
1. The String Matching Problem

Given text T (length n) and pattern P (length m), find all positions i where T[i..i+m−1] = P. Applications: text editors (Ctrl+F), bioinformatics (DNA search), network intrusion detection, plagiarism detection.

2. Why Rolling Hash?

Naïve comparison rechecks every character in every window — O(n×m). The key insight: if we could compare windows in O(1), the total becomes O(n). Hashing achieves this — but we can't afford to recompute the hash from scratch each time. Rolling hash reuses the previous window's hash, updating in O(1).

3. The Hash Formula
hash(S[0..m-1]) = (S[0]×d^(m-1) + S[1]×d^(m-2) + … + S[m-1]) mod q // Precompute: h = d^(m-1) mod q // Rolling update — slide window right by 1: hash_new = (d × (hash_old - T[i] × h) + T[i+m]) mod q // Add q before mod to handle negative values: hash_new = (d × (hash_old - T[i] × h + q) + T[i+m]) mod q
4. Spurious Hits — Why We Must Verify

Two different strings can have the same hash (collision). A spurious hit is when hash(window) = hash(P) but window ≠ P. When this happens, we do a full O(m) character-by-character check. The key: spurious hits are rare with good q, so expected total work is still O(n+m).

5. Full Algorithm
RABIN-KARP(T, P, d, q): n = len(T); m = len(P) h = d^(m-1) mod q // high-order coefficient pHash = 0; wHash = 0 for i = 0 to m-1: // compute initial hashes pHash = (d*pHash + P[i]) mod q wHash = (d*wHash + T[i]) mod q for i = 0 to n-m: // slide window if pHash == wHash: // hash match if T[i..i+m-1] == P: // verify characters report match at i if i < n-m: // roll hash wHash = (d*(wHash - T[i]*h) + T[i+m]) mod q wHash = (wHash + q) mod q // handle negatives
6. Complexity Analysis

Preprocessing: O(m) to compute initial hashes. Each slide: O(1) hash update. Each hash match: O(m) verification (rare). Expected total: O(n + m). Worst case (all windows hash-match): O(nm). With large prime q, collisions are rare so expected case dominates in practice.

7. Choosing d and q

d = alphabet size (26 for lowercase letters, 256 for ASCII). q = a large prime (101, 1009, 1000003). The probability of a spurious hit at any position is approximately 1/q. With q=101 and a text of length 1000, expect about 10 spurious hits. With q=1000003, expect about 0.001 — essentially none.
